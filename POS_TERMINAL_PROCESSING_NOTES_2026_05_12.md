# POS Terminal / Processing Notes - 2026-05-12

## Контекст

Нужно принимать оплату с пульта/CRM через банковский терминал в студии Соборный.

Текущая рабочая цепочка в проекте уже рассчитана на POS-агент:

```text
CRM / пульт
  -> backend /api/pos/bridge/pay
  -> pos_transactions
  -> print-api / MQTT
  -> SvfPosAgent на ПК Соборного
  -> INPAS SmartSale / DualConnector
  -> PAX D230
```

Проблема: хочется убрать зависимость от нестабильного ПК и использовать терминал напрямую через сеть/роутер.

## Что выяснили по текущему D230

- Терминал: PAX D230.
- Сам факт, что D230 подключен к Wi-Fi/роутеру, не означает, что CRM может отправлять на него сумму оплаты.
- Wi-Fi на терминале часто нужен только для связи терминала с банком.
- Для прямой интеграции нужен один из режимов:
  - прямой ECR TCP/IP на IP-адрес терминала;
  - POSLink / аналогичный протокол;
  - SDK/API на самом терминале;
  - облачный API банка/процессинга по terminal_id.
- DualConnector для нас является локальным шлюзом/сервером на Windows-ПК:

```text
CRM / agent -> DualConnector на ПК -> терминал -> банк
```

Если банк говорит, что DualConnector обязателен, значит полностью обойти ПК нельзя.

## Альфа-Банк

Из вариантов Альфы по типу продукта под нашу задачу подходит только:

```text
Интегрируемый терминал
```

Потому что по описанию он нужен для передачи суммы с кассы/CRM на терминал.

Не подходят как основной путь:

- "Касса 3 в 1" - это их касса вместо нашей системы.
- "Переносной терминал" - обычно ручной сценарий.
- "AlfaPOS" / "AlfaPOS и AlfaCASH" - Android-приложения Альфы вместо нашей кассы/терминала.

Ответ Альфы по PAX S300:

```text
PAX S300 - интеграционный терминал.
Не имеет встроенного веб-сервера, не реализует HTTP/API,
не отвечает на прямые TCP-запросы от внешних систем.
DualConnector обязателен, обойти его не получится.
```

Вывод: PAX S300 от Альфы не решает задачу обхода ПК, если они действительно дают только INPAS SmartSale + DualConnector.

## Как правильно спрашивать банки

Не спрашивать про "веб-сервер" как главное требование. Встроенный веб-сервер не обязателен.

Нужно спрашивать про режим интеграции:

```text
Поддерживает ли ваш терминал прямую кассовую интеграцию без DualConnector на ПК?
Нужен один из вариантов: ECR Comm. Type Ethernet/TCP-IP, POSLink,
SDK/API на терминале или облачный API по terminal_id.
```

Короткая формулировка:

```text
Есть ли у ваших терминалов режим прямой интеграции с кассовым ПО
по ECR Ethernet/TCP-IP или POSLink без DualConnector?
```

Если банк не знает прошивку заранее:

```text
Можно ли при заказе зафиксировать обязательное требование:
прямая интеграция с нашей CRM без ПК-посредника?
Если после получения терминала такого режима нет, можно ли заменить терминал
или отказаться от него?
```

## ECR Comm. Type

Корректный термин из документации PAX:

```text
ECR Comm. Type -> Ethernet -> TCP/IP
```

ECR здесь означает кассу/кассовое ПО.

Важно: наличие такого режима в базовой документации PAX не гарантирует, что банковская прошивка/платежное приложение его разрешает.

Поэтому правильный вопрос:

```text
Поддерживает ли ваша прошивка/платежное приложение на PAX режим
ECR Comm. Type: Ethernet -> TCP/IP без DualConnector?
```

## Свой терминал и процессинговый центр

Идея: купить свой терминал и заключить договор напрямую с процессинговым центром.

Это возможно только если процессинг поддерживает merchant-owned terminals, то есть подключение терминалов клиента.

Одного MID и расчетного счета недостаточно. Для card-present оплаты нужны:

- MID;
- TID для конкретного терминала;
- регистрация серийника терминала;
- платежное приложение;
- TMS/host-профиль;
- загрузка ключей;
- маршрутизация на процессинг;
- поддерживаемый протокол интеграции с CRM.

Что спрашивать у процессинга:

```text
Поддерживаете ли вы подключение собственного POS-терминала?
Какие модели и платежные приложения вы поддерживаете?
Можете ли зарегистрировать наш терминал, загрузить ключи/TMS-профиль
и выдать TID?
Нужна интеграция с нашей CRM без ПК-посредника:
ECR TCP/IP, POSLink, Android SDK/API или cloud API.
```

## Важен ли город процессингового центра

Город обычно не принципиален.

Для процессинга важнее:

- поддерживают ли они собственные терминалы;
- какие модели/прошивки принимают;
- есть ли прямой протокол/API без ПК;
- как они обслуживают терминалы;
- как быстро меняют/ремонтируют оборудование;
- есть ли выездная поддержка в Ростове-на-Дону.

То есть процессинг может быть не в Ростове, если он умеет подключить нужную схему и нормально обслуживать терминалы.

Город важен только для практики:

- доставка;
- замена терминала;
- сервисный выезд;
- срочный ремонт;
- наличие локального партнера.

   - подключение собственного терминала.

## Короткий вывод

Если банк/процессинг говорит "только DualConnector", то без какого-либо ПК/Windows-шлюза не получится.

Если нужен обход ПК, искать нужно не "PAX D230/S300", а конкретную схему:

```text
direct ECR TCP/IP / POSLink / Android SDK / cloud API
```

или процессинг, который подключает собственные терминалы и дает поддерживаемый протокол интеграции.
